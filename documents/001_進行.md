0. 導入…Next.js v16ではオプションがあったはず
   0.1 create 時に Biome を選べるか（対話式 / フラグ）を確認
   0.2 生成物（biome.json など）の初期値をメモ

1. 一般的な React / Next.js プロジェクトにおける ESLint / Prettier の設定例を整理する
   1.1 代表的な .eslintrc / .prettierrc の最小〜中量構成を2〜3通り用意
   1.2 目的（可読性・安全性・チーム運用）ごとに「何を効かせるか」を明文化

2. Biome のデフォルト設定
   2.1 確認（ゼロ設定時の挙動／recommended の範囲）
   2.2 Next.js / React に対してどのようなルールを持っているか把握する
   2.3 デフォルト設定と、一般的に用いられる ESLint / Prettier 設定との違いを比較する
   2.4 差分が大きい領域（import順序・未使用検出・JSX規約・整形幅など）を洗い出し

3. 必要に応じて Biome 上で ESLint / Prettier の設定を上書きする方法を確認する
   3.1 biome.json / biome.jsonc のキー構成を把握（formatter / linter / files など）
   3.2 上書きで再現したい ESLint/Prettier の振る舞いを個別に移植

4. 実際に動作させながら、設定変更がコード整形やコード検証にどのような影響を与えるか確認する (lint/format)
   4.1 小さな検証用リポジトリで format / lint を実行
   4.2 差分（git で確認）を記録し、採用/非採用の判断基準を決める

5. 移行検証（ESLint / Prettier → Biome）
   5.1 既存の ESLint / Prettier 構成を一旦適用した Next.js プロジェクトを用意
   5.2 Biome を導入してルール対応状況を比較（未移植・非対応箇所を列挙）
   5.3 競合回避（VS Code / npm scripts / pre-commit での実行順）を整理
   5.4 段階移行のガイド（まず formatter、次に linter など）を下書き

6. 初期化体験の比較
   6.1 `biome init` 相当の対話式初期化を試す
   6.2 Next.js v16 のテンプレート生成時に作られる設定との違いを記録

7. VS Code 拡張の設定
   7.1 保存時整形、診断の有効化、ワークスペース設定の最小例を作成
   7.2 既存の ESLint / Prettier プラグインとの干渉点を確認

8. CI（GitHub Actions）での最小実行
   8.1 format チェック（差分検知）と lint のジョブを作成
   8.2 ワークスペース内キャッシュや失敗時のログ出力ポリシーを決める

9. ドキュメント確認の観点
   9.1 「VCS＝バージョン管理システム（git等）」の統合手順
   9.2 移行ガイド（ESLint / Prettier からの置き換え）
   9.3 設定リファレンス（キーと既定値、上書きの優先順）
